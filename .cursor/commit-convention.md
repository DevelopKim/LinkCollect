# 커밋 메시지 규칙

## 형식
```
<type>: <subject>

<body>
```

## Type 종류

- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 작성/수정 (README, PRD, TRD 등)
- `style`: 코드 포맷팅, 세미콜론 누락 등 (코드 변경 없음)
- `refactor`: 코드 리팩토링
- `test`: 테스트 코드 추가/수정
- `chore`: 빌드 업무 수정, 패키지 매니저 설정 등
- `build`: 빌드 시스템 수정
- `perf`: 성능 개선

## Subject 작성 규칙

- 첫 글자는 대문자로 시작하지 않음
- 마침표(.)를 붙이지 않음
- 명령형으로 작성 (과거형 X)
- 50자 이내로 간결하게

## Body 작성 규칙 (선택사항)

- 변경 사항을 상세히 설명
- 무엇을 변경했는지, 왜 변경했는지 설명
- 관련 이슈 번호가 있다면 참조
- 각 줄은 72자 이내

## 예시

### 문서 작성
```
docs: 프로젝트 문서 작성 (PRD, TRD)

- PRD (Product Requirements Document) 작성
- TRD (Technical Requirements Document) 작성
- 프로젝트 요구사항 및 기술 명세 정리
```

### 기능 추가
```
feat: 링크 추출 기능 구현

- Content Script를 통한 DOM에서 링크 추출
- 도메인 필터링 로직 구현
- 추출된 링크를 Popup에 전송하는 메시지 통신 구현
```

### 버그 수정
```
fix: 상대 경로를 절대 URL로 변환하는 로직 수정

- base URL 처리 로직 개선
- edge case 처리 추가
```

### 리팩토링
```
refactor: 포맷 시스템을 인터페이스 기반으로 리팩토링

- IFormat 인터페이스 정의
- CSV 포맷을 클래스로 분리
- 확장 가능한 구조로 개선
```

## 참고사항

- 여러 변경사항이 있다면 각각 별도 커밋으로 분리하는 것을 권장
- 의미 있는 단위로 커밋 (너무 작지도 크지도 않게)
- 커밋 전에 변경사항을 다시 확인
